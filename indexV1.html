02 Stack Técnico

Cuatro pilares.
Una app robusta.

Cada librería elegida cumple un rol específico — nada por moda, todo por necesidad operativa.

01 / 04
ROOM
Persistencia local · SQLite

Base de datos embebida en el dispositivo. Capa de abstracción sobre SQLite que permite trabajar con clases Java en lugar de SQL puro. Validación de consultas en tiempo de compilación.

v 2.6.1 · Offline-first
02 / 04
HILT
Inyección de dependencias

Construido sobre Dagger 2. Resuelve y conecta automáticamente las clases del proyecto. Garantiza que la base de datos sea Singleton — una única instancia para toda la app.

v 2.51.1 · @HiltAndroidApp
03 / 04
RETROFIT
Cliente HTTP · APIs REST

Preparado para futuras integraciones — consulta de datos meteorológicos en tiempo real (Open-Meteo) sobre la zona de intervención, sincronización con servidor central en versiones siguientes.

v 2.9.0 · Roadmap v1.1
04 / 04
MAPS · GPS
Geolocalización · UTM

Google Maps SDK con vista satélite optimizada para identificación de zonas forestales. Sistema UTM Huso 30T — el formato estándar usado por servicios de emergencia en Aragón.

v 18.1.0 · Huso 30T
03 Arquitectura · Patrón MVVM

Tres capas.
Cero acoplamiento.

CAPA 01 · UI
VISTA

↓
CAPA 02 · LOGIC
LÓGICA

CAPA 03 · DATA
DATOS
VehiculoRepository · DAO · AppDatabase
Room SQLite + ExecutorService

¿Por qué tres capas?

La separación de responsabilidades es la clave para que el código pueda crecer sin convertirse en un caos. Cada capa tiene un trabajo, y solo uno.

Si mañana se cambia el diseño visual, solo se toca la Vista. Si se sustituye SQLite por un servidor en la nube, solo se modifica la capa de Datos. La Lógica permanece intacta.

El Repository actúa como guardián entre la Lógica y los Datos: las pantallas nunca acceden directamente a la base de datos. Pasan siempre por él.

Esta estructura es la base recomendada por Google para apps Android de calidad profesional.
